# 调试与问题解决工作流

这是一个系统化的问题诊断和解决工作流，整合了 systematic-debugging、verification-before-completion、test-driven-development 等 skills，确保问题得到根本性解决。

## 触发场景

当用户提到以下需求时自动触发：
- "报错了"、"出 bug 了"、"不行了"
- "帮我看看这个问题"、"为什么 XXX"
- "调试一下"、"修复这个错误"
- "性能慢"、"内存泄漏"
- "连接失败"、"认证错误"

## 完整工作流程

### 阶段 1：问题描述与信息收集

**1.1 问题澄清**
```
必需信息：
- 错误信息（完整堆栈）
- 复现步骤
- 预期行为 vs 实际行为
- 环境信息（OS、版本等）
- 发生时间/频率
```

**1.2 上下文收集**
```
动作：
- 读取相关代码文件
- 查看日志输出
- 检查配置文件
- 了解最近变更
```

### 阶段 2：问题复现

**2.1 创建复现环境**
```
动作：
- 搭建最小复现场景
- 隔离外部依赖
- 确保稳定复现
```

**2.2 记录复现步骤**
```
产出：
- 详细复现步骤文档
- 复现脚本（如适用）
- 截图/录屏（如适用）
```

### 阶段 3：系统调试（核心）

**调用：systematic-debugging skill**

**3.1 阶段 1：问题定位**
```
目标：缩小问题范围

方法：
- 二分法定位（如适用）
- 日志分析
- 断点调试
- 请求追踪

产出：问题所在的模块/组件
```

**3.2 阶段 2：根因分析**
```
目标：找到根本原因

方法：
- 5 Why 分析法
- 鱼骨图分析
- 对比分析（正常 vs 异常）
- 假设验证

产出：根因报告
```

**3.3 阶段 3：方案设计**
```
目标：设计修复方案

考虑：
- 临时缓解措施
- 根本性修复
- 长期预防方案

产出：修复方案文档
```

**3.4 阶段 4：修复验证**
```
目标：验证修复有效

方法：
- 单元测试
- 集成测试
- 回归测试
- 性能测试（如适用）

产出：验证报告
```

### 阶段 4：修复实施

**4.1 TDD 修复**
```
调用：test-driven-development skill

流程：
1. 写一个暴露问题的失败测试
2. 观察测试失败（确认问题存在）
3. 写最少代码修复问题
4. 观察测试通过
5. 重构优化
6. 提交代码
```

**4.2 代码审查**
```
调用：requesting-code-review skill

审查重点：
- 修复是否正确
- 有无副作用
- 边界情况处理
- 性能影响
```

### 阶段 5：验证测试

**5.1 完成验证**
```
调用：verification-before-completion skill

检查清单：
✓ 所有测试通过
✓ 问题不再复现
✓ 无新回归问题
✓ 性能符合预期
✓ 文档已更新
```

**5.2 边界测试**
```
测试：
- 边界条件
- 异常输入
- 并发场景（如适用）
- 长时间运行（如适用）
```

### 阶段 6：预防措施

**6.1 根因消除**
```
动作：
- 更新相关代码
- 添加防御性检查
- 完善错误处理
```

**6.2 文档更新**
```
更新：
- README 常见问题
- 故障排查手册
- 最佳实践文档
```

**6.3 经验总结**
```
产出：
- 问题分析报告
- 修复过程记录
- 预防措施清单
```

## 快捷命令模式

```bash
# 完整流程
/debug-troubleshoot-workflow "登录后 500 错误"

# 仅诊断
/debug-troubleshoot-workflow <问题> --stage diagnose

# 仅修复
/debug-troubleshoot-workflow <根因> --stage fix

# 紧急修复（跳过部分验证）
/debug-troubleshoot-workflow <问题> --mode emergency

# 性能问题专项
/debug-troubleshoot-workflow "接口响应慢" --type performance

# 内存问题专项
/debug-troubleshoot-workflow "内存泄漏" --type memory

# 并发问题专项
/debug-troubleshoot-workflow "竞态条件" --type concurrency
```

## 问题分类与专项流程

### Bug 修复

```
特点：功能异常、报错
流程：标准 6 阶段
重点：根因分析、回归测试
```

### 性能优化

```
特点：慢、卡、资源占用高
额外步骤：
- 性能分析（profiling）
- 瓶颈识别
- 优化方案对比
- 性能回归测试
```

### 内存问题

```
特点：内存泄漏、OOM
额外步骤：
- 内存快照分析
- 对象引用追踪
- GC 日志分析
```

### 并发问题

```
特点：竞态条件、死锁
额外步骤：
- 并发场景复现
- 锁分析
- 线程转储分析
```

### 网络问题

```
特点：连接失败、超时
额外步骤：
- 网络抓包
- DNS/路由追踪
- 防火墙检查
```

## 诊断工具箱

### 日志分析
```bash
# 查看应用日志
tail -f logs/app.log

# 搜索错误
grep -i "error\|exception" logs/*.log

# 时间范围筛选
grep "2026-04-28 1[0-2]:" logs/app.log
```

### 系统状态
```bash
# CPU/内存
top -o %CPU

# 磁盘
df -h

# 网络
netstat -tulpn

# 进程
ps aux | grep <process>
```

### 代码级诊断
```
- 断点调试
- 代码覆盖率
- 静态分析
- 依赖检查
```

## 使用示例

### 示例 1：API 报错 500

用户："用户登录接口返回 500 错误"

工作流：
1. 收集信息：错误日志、请求参数、堆栈信息
2. 复现问题：用相同参数调用接口
3. systematic-debugging：
   - 定位：认证模块
   - 根因：空指针异常
   - 方案：添加空值检查
   - 验证：测试通过
4. TDD 修复：写测试→修复→验证
5. verification-before-completion：全量回归
6. 更新文档：添加到常见问题

### 示例 2：内存泄漏

用户："服务运行几天后内存爆满"

工作流：
1. 收集信息：内存使用曲线、GC 日志
2. 复现：压力测试模拟
3. systematic-debugging：
   - 定位：用户会话管理
   - 根因：会话未超时释放
   - 方案：添加 TTL 和定期清理
4. 实施修复
5. 长时间运行验证
6. 添加内存监控告警

### 示例 3：性能慢

用户："查询接口响应超过 5 秒"

工作流：
1. 收集信息：慢查询日志、执行计划
2. 复现：相同查询条件
3. systematic-debugging：
   - 定位：数据库查询
   - 根因：缺少索引
   - 方案：添加复合索引
4. 实施修复
5. 性能对比测试（修复前 vs 修复后）
6. 添加性能监控

## 紧急模式

```bash
--mode emergency
```

用于生产事故，流程简化：
1. 快速定位（15 分钟内）
2. 临时缓解（先恢复服务）
3. 根本修复（后续跟进）
4. 事后复盘（必须）

## 进阶配置

创建 `.debug-workflow/EXTEND.md`：

```yaml
# 调试偏好
debugging:
  prefer_tool: pdb  # 或 breakpoint、logging
  log_level: DEBUG
  max_repro_time: 30min  # 复现时间上限

# 验证要求
verification:
  min_test_coverage: 90%
  regression_required: true
  performance_budget:
    response_time: <100ms
    memory_increase: <10%

# 报告模板
report:
  include_sections:
    - problem_description
    - reproduction_steps
    - root_cause
    - fix_implementation
    - verification_results
    - prevention_measures
```

## 最佳实践

1. **先复现再修复**：确保能稳定复现问题
2. **一次只改一处**：避免多个变更混淆结果
3. **测试先行**：先写暴露问题的测试
4. **根因导向**：不要只修复症状
5. **文档沉淀**：每次问题都更新知识库

## 注意事项

1. **生产优先**：生产问题先恢复再分析
2. **数据保护**：调试时注意数据安全
3. **变更记录**：所有修改都要记录
4. **回滚准备**：修复前准备回滚方案
5. **及时同步**：重要问题及时同步团队
