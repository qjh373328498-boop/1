# 方案 D 增强版：高精度意图识别方案

## 一、核心问题分析

### 1.1 可能的错误场景

| 错误类型 | 示例 | 后果 |
|---------|------|------|
| **漏触发** | 用户说"下周要考中级会计了" → 未识别出需要 exam-prep | 用户得不到专业帮助 |
| **误触发** | 用户说"我想了解一下这个比赛" → 触发 competition-prep（其实只是好奇） | 浪费 token，用户体验差 |
| **错误触发** | 用户说"分析这个销售数据" → 触发 exam-prep（应该是 data-analysis） | 执行错误工作流 |
| **部分触发** | 用户说"写论文需要查文献和数据分析" → 只触发 research-paper | 遗漏部分需求 |

---

## 二、解决方案：多层意图识别系统

### 2.1 架构设计

```
用户输入
    ↓
┌─────────────────────────────────┐
│  第 1 层：粗筛（关键词匹配）       │
│  快速排除无关输入                │
│  输出：候选技能列表 (1-N 个)     │
└──────────────┬──────────────────┘
               ↓
┌─────────────────────────────────┐
│  第 2 层：精筛（语义分析）         │
│  上下文理解、否定词识别          │
│  输出：排序后的技能列表          │
└──────────────┬──────────────────┘
               ↓
┌─────────────────────────────────┐
│  第 3 层：置信度评估）            │
│  计算匹配置信度 (0-100%)         │
│  输出：是否触发 + 置信度         │
└──────────────┬──────────────────┘
               ↓
┌─────────────────────────────────┐
│  第 4 层：用户确认（可选）         │
│  低置信度时询问用户              │
│  输出：最终确认的技能            │
└─────────────────────────────────┘
```

---

### 2.2 第 1 层：粗筛（关键词匹配）

**目的**：快速筛选，宁可多选不可漏选

**实现**：
```markdown
## 多级关键词体系

### 一级关键词（强匹配）
- "写论文" → research-paper (权重 100)
- "复习" → exam-prep (权重 100)
- "比赛" → competition-prep (权重 100)

### 二级关键词（中匹配）
- "论文" → research-paper (权重 70)
- "考试" → exam-prep (权重 70)
- "竞赛" → competition-prep (权重 70)

### 三级关键词（弱匹配）
- "文献" → literature-search (权重 50)
- "数据" → data-analysis (权重 50)
- "计划" → weekly-planning (权重 50)

### 关键词组合
- "论文" + "写" → research-paper (权重 80)
- "考试" + "准备" → exam-prep (权重 80)
- "数据" + "分析" → data-analysis (权重 80)
```

**粗筛策略**：
- 权重 ≥ 80：直接进入第 3 层
- 权重 50-79：进入第 2 层精筛
- 权重 < 50：排除

---

### 2.3 第 2 层：精筛（语义分析）

**目的**：识别真实意图，排除干扰

#### 2.3.1 否定词识别

```markdown
## 否定词列表

### 明确否定
- 不、不要、不想、不需要、别、甭
- 没有、还没、尚未

### 疑问否定
- 怎么不、为什么不、何必

### 示例
❌ "我不想写论文" → 不触发
❌ "怎么不考试" → 不触发
❌ "还没有开始复习" → 不触发（但可能是需求信号）
```

#### 2.3.2 意图强度识别

```markdown
## 强意图信号（直接触发）
- 帮我 XXX
- 我要 XXX
- 需要 XXX
- 怎么做 XXX
- 准备 XXX

## 弱意图信号（需要确认）
- 了解一下 XXX
- 看看 XXX
- 问一下 XXX
- XXX 是什么

## 无意图信号（不触发）
- XXX 怎么样
- XXX 好不好
- 知道 XXX 吗
```

#### 2.3.3 上下文理解

```markdown
## 结合上文判断

### 示例 1
用户："下周有考试"
模型："什么科目？需要我帮你制定复习计划吗？"
用户："中级会计"
→ 此时触发 exam-prep（上下文中"考试"+"复习计划"）

### 示例 2
用户："学创杯什么时候报名？"
模型："通常在 3 月，你是想参加吗？"
用户："对，想了解一下"
→ 此时不触发 competition-prep（只是咨询，不是备赛）

### 示例 3
用户："学创杯什么时候报名？"
模型："通常在 3 月，你是想参加吗？"
用户："对，帮我准备"
→ 此时触发 competition-prep
```

### 2.4 第 3 层：置信度评估

**目的**：量化匹配程度，决定是否触发

#### 置信度计算公式

```
置信度 = 基础权重 × 意图系数 × 上下文系数

基础权重：
- 一级关键词匹配：100%
- 二级关键词匹配：70%
- 三级关键词 + 组合：50-80%

意图系数：
- 强意图信号：× 1.0
- 弱意图信号：× 0.6
- 否定信号：× 0（不触发）

上下文系数：
- 上文有相关讨论：× 1.2
- 首次提及：× 1.0
- 上文无关：× 0.8
```

#### 置信度阈值

| 置信度 | 决策 | 操作 |
|--------|------|------|
| ≥ 80% | 高置信 | 直接触发 skill |
| 60-79% | 中置信 | 触发 + 简要告知用户 |
| 40-59% | 低置信 | 询问用户确认 |
| < 40% | 不触发 | 正常对话 |

---

### 2.5 第 4 层：用户确认（可选）

**目的**：低置信度时让用户决策

#### 确认话术模板

```markdown
## 高置信度告知（≥80%）

"好的，我来帮你{任务}，我将：
1. {步骤 1}
2. {步骤 2}
3. {步骤 3}

准备好了吗？"
```

```markdown
## 中置信度确认（60-79%）

"我理解你想{任务}，我会使用{技能名}来帮你，包括：
- {功能 1}
- {功能 2}

这样可以吗？"
```

```markdown
## 低置信度询问（40-59%）

"我想确认一下，你是想：

A. {选项 A：比赛备赛} - 制定备赛计划、准备材料
B. {选项 B：了解比赛} - 查看比赛规则、时间
C. {选项 C：其他需求}

请告诉我你的具体需求~"
```

```markdown
## 不确定时的通用询问

"我想确认一下你的需求：

{用户输入}

你是想：
1. {最可能的技能 A}
2. {次可能的技能 B}
3. 其他需求（请详细描述）"
```

## 三、实战示例

### 3.1 正确触发场景

#### 场景 1：明确需求
```
用户："帮我复习中级会计，下周三考试"

识别过程：
1. 粗筛：
   - "复习" → exam-prep (权重 100) ✓
   - "考试" → exam-prep (权重 70) ✓
   
2. 精筛：
   - "帮我" → 强意图信号 ✓
   - 无否定词 ✓
   
3. 置信度：
   - 基础权重：100%
   - 意图系数：× 1.0
   - 上下文系数：× 1.0
   - 最终：100% ✓

决策：直接触发 exam-prep
响应："好的，我来帮你准备中级会计考试。我先了解一下：
1. 你有课程笔记或 PPT 吗？
2. 已经复习到什么程度了？"
```

#### 场景 2：多意图识别
```
用户："写论文需要查文献和数据分析"

识别过程：
1. 粗筛：
   - "写论文" → research-paper (100) ✓
   - "查文献" → literature-search (100) ✓
   - "数据分析" → data-analysis (100) ✓
   
2. 精筛：
   - 无否定词 ✓
   - 陈述需求 ✓
   
3. 置信度：
   - 三个技能都是 100% ✓

决策：触发 3 个 skills
响应："好的，我来帮你完成论文写作，包括：
1. 文献检索（literature-search）
2. 数据分析（data-analysis）
3. 论文写作（research-paper）

我们按顺序进行，先处理哪个？"
```

#### 场景 3：上下文理解
```
用户（第 1 轮）："下周有考试"
模型："什么科目？需要我帮你制定复习计划吗？"
用户（第 2 轮）："中级会计"

识别过程：
1. 粗筛：
   - "中级会计" → 课程名 → exam-prep (70)
   
2. 精筛：
   - 上文提到"考试"+"复习计划" ✓
   - 上下文系数：× 1.2
   
3. 置信度：
   - 70% × 1.2 = 84% ✓

决策：触发 exam-prep
响应："好的，中级会计的复习计划。你有课程笔记或 PPT 吗？"
```

### 3.2 避免误触发场景

#### 场景 1：否定句
```
用户："我不想写论文，太累了"

识别过程：
1. 粗筛：
   - "写论文" → research-paper (100)
   
2. 精筛：
   - "不想" → 否定词 ✗
   - 意图系数：× 0
   
3. 置信度：
   - 100% × 0 = 0%

决策：不触发
响应："理解，写论文确实很辛苦。那你想聊点什么？"
```

#### 场景 2：仅咨询
```
用户："学创杯是什么比赛？"

识别过程：
1. 粗筛：
   - "学创杯" → competition-prep (70)
   
2. 精筛：
   - "是什么" → 疑问，非意图信号
   - 意图系数：× 0.5
   
3. 置信度：
   - 70% × 0.5 = 35%

决策：不触发（< 40%）
响应："学创杯是...'（正常回答，不启动工作流）"
```

#### 场景 3：好奇心 Query
```
用户："数据分析怎么做？"

识别过程：
1. 粗筛：
   - "数据分析" → data-analysis (50)
   - "怎么做" → 疑问
   
2. 精筛：
   - 意图系数：× 0.6
   
3. 置信度：
   - 50% × 0.6 = 30%

决策：不触发
响应："数据分析的步骤是...'（先简单解释）
想让我帮你分析具体数据吗？"（反问确认）
```

### 3.3 需要确认的场景

#### 场景 1：模糊需求
```
用户："帮我看看这个比赛"

识别过程：
1. 粗筛：
   - "比赛" → competition-prep (70)
   - "帮我" → 强意图信号
   
2. 精筛：
   - "看看" → 弱意图
   - 意图系数：× 0.6
   
3. 置信度：
   - 70% × 0.6 = 42%

决策：询问确认
响应："这个比赛是？另外我想确认一下，你是想：

A. 了解比赛规则和截止时间
B. 准备参赛（备赛计划、材料准备）
C. 其他需求

请告诉我~"
```

#### 场景 2：多技能边界
```
用户："我要做个数据相关的作业"

识别过程：
1. 粗筛：
   - "数据" → data-analysis (50)
   - "作业" → exam-prep (40)
   
2. 精筛：
   - 两个技能置信度都偏低
   
3. 置信度：
   - data-analysis: 50%
   - exam-prep: 40%

决策：询问确认
响应："数据相关的作业，具体是什么类型？

A. 数据分析（用 Python/Excel 处理数据、生成图表）
B. 数据相关的理论知识复习
C. 其他类型

请详细描述一下~"
```

## 七、实施建议

```
- 添加否定词识别

```

### 阶段 3：上下文理解（准确率~90%）
```
- 对话历史分析
- 多轮对话状态追踪
- 上下文系数调整
```

### 阶段 4：机器学习（准确率~95%）
```
- 收集用户反馈

- 个性化适配
```

## 八、决策点

### 精度 vs 速度 权衡

| 配置 | 准确率 | 确认频率 | 用户体验 |
|------|--------|---------|---------|
| 激进模式 | 75% | 低 | 响应快，但容易错 |
| 平衡模式 | 90% | 中 | 推荐 ✅ |
|保守模式 | 95% | 高 | 准确但繁琐 |

**推荐配置**：
- 直接触发阈值：80%
- 确认触发阈值：60%
- 询问阈值：40%

## 九、总结

### 方案 D 增强版的核心优势

1. **多层过滤**：避免单一关键词误判
2. **置信度量化**：明确何时触发/确认/不触发
3. **上下文理解**：结合对话历史判断
4. **用户确认机制**：低置信度时让用户决策
5. **兜底方案**：支持手动指定技能

### 预期效果

- ✅ 漏触发率 < 10%
- ✅ 误触发率 < 10%
- ✅ 综合准确率 > 90%
- ✅ Token 节省 85%+
